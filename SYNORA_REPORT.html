1. INTRODUCTION OF THE PROJECT


1.1 Background of the Project


The rapid advancement of digital health technologies and Artificial Intelligence has created an unprecedented opportunity to transform traditional healthcare delivery systems. In India, the healthcare sector faces significant challenges including insufficient doctor-patient ratios, fragmented medical record systems, inaccessible healthcare in rural areas, and lack of real-time health monitoring solutions. Patients struggle to book appointments, access their medical history, or get timely medical advice, while doctors face difficulties in managing their schedules, patient data, and consultations efficiently.


The "Synora Health" platform is developed to address these challenges by building a comprehensive AI-powered healthcare management ecosystem. The system integrates modern web technologies with advanced Artificial Intelligence and Large Language Models (LLMs) to provide a seamless experience for patients, doctors, and administrators in a single unified platform.


The project was developed during a 6-month internship at Coding House, Surat, Gujarat, using a modern technology stack comprising Python FastAPI for the backend, React 18 with Vite for the frontend, MongoDB for data storage, and AI engines including Groq (Llama 3.1), Google Gemini, and OpenAI GPT-4 for intelligent health analysis.


Synora Health serves three primary user roles — Patients, Doctors, and Administrators — each with a dedicated dashboard and feature set. The platform enables patients to book appointments, consult doctors via video calls, track health records, get AI-powered symptom analysis, manage medicine reminders, and monitor health goals. Doctors can manage their availability, view patient histories, write prescriptions, and conduct real-time consultations. Administrators maintain platform integrity by verifying doctors, managing users, and analyzing system performance through intelligent dashboards.

1.2 Problem Statement


In the current healthcare environment, both patients and medical professionals face numerous challenges due to the absence of an integrated, intelligent digital platform. The major problems identified are:


  • Patients must visit multiple platforms separately for appointment booking, medical records, prescription management, and health monitoring.
  • Manual appointment systems lead to scheduling conflicts, long waiting times, and poor patient experience.
  • Medical records are stored in paper format or scattered across different systems, making it difficult to maintain a unified patient health history.
  • Doctors lack AI-powered tools to assist in diagnosis, symptom analysis, and patient risk evaluation during consultations.
  • Rural and semi-urban patients have limited access to specialist doctors due to geographic barriers, with no telemedicine solution available.
  • Medicine adherence is poor due to the absence of intelligent reminder systems linked to digital calendars.
  • Insurance claim management is manual, time-consuming, and lacks transparency for both patients and insurers.
  • Emergency situations require instant access to medical assistance, which is not available through existing fragmented systems.
  • Healthcare administrators lack real-time analytics and management tools to oversee platform activity, verify medical professionals, and monitor system health.
  • Existing platforms do not integrate AI-based health analysis, real-time consultation, payment gateways, and comprehensive health tracking into a single ecosystem.

These limitations lead to delayed diagnoses, poor treatment adherence, inefficient healthcare delivery, and overall degraded patient outcomes. The proposed Synora Health platform directly addresses all these issues through a unified, AI-enhanced digital healthcare ecosystem.

Detailed Description of Key Technologies


1. Python & FastAPI

Python serves as the core backend language. FastAPI provides high-performance, async-first REST API development with automatic OpenAPI documentation, Pydantic validation, and dependency injection. Its async capabilities using Motor driver enable non-blocking database operations.


2. MongoDB (Motor – Async Driver)

MongoDB is used as the document-oriented NoSQL database. Motor provides fully asynchronous access to MongoDB, enabling concurrent request handling. Collections include users, doctors, appointments, health_records, prescriptions, messages, payments, and more.


3. React 18 + Vite + Tailwind CSS

React 18 with concurrent rendering provides a responsive, component-based UI. Vite offers ultra-fast build times. Tailwind CSS enables rapid, utility-first UI development with professional styling. Redux Toolkit manages global state including authentication tokens and appointment data.


4. Groq API (Llama 3.1-70B)

Groq's ultra-fast inference engine powers the AI Health Copilot chatbot and symptom analysis. The Llama 3.1-70B model provides accurate medical reasoning, symptom assessment, and health guidance in real time.


5. Google Gemini 1.5 Pro

Google's Gemini model is used for medical image and report analysis, enabling the Smart Report Analyzer feature that extracts health parameters from uploaded medical documents.


6. JWT Authentication + Twilio SMS OTP

JSON Web Tokens provide stateless authentication with access and refresh token management. Twilio SMS delivers 6-digit OTP codes for secure email verification during registration, with email fallback for users without phone numbers.


7. Razorpay Payment Gateway

Razorpay handles appointment fee collection with HMAC-SHA256 signature verification, supporting UPI, credit/debit cards, and net banking. A demo payment mode is available for testing environments.

Existing healthcare digital platforms in India primarily offer isolated services without intelligent integration. Key limitations of existing systems include:


  • Fragmentation: Patients must use multiple apps for appointments, records, consultations, and health tracking.
  • No Real AI Integration: Most platforms use rule-based chatbots rather than true LLM-powered health analysis.
  • Limited Telemedicine: Video consultation features are often premium-only or unreliable.
  • No Insurance Integration: Claims management and insurance tracking are not integrated.
  • No Preventive Health Tools: Medicine reminders, health goals, and symptom diaries are rarely available in integrated form.
  • Poor Doctor Experience: Doctor-facing tools are limited, lacking AI-assisted diagnosis support.
  • No Smart Report Analysis: Medical documents cannot be automatically analyzed and interpreted.

The proposed Synora Health platform overcomes all these limitations by delivering a fully integrated, AI-powered healthcare ecosystem that serves all stakeholders — patients, doctors, and administrators — through a single, cohesive platform with modern technology standards.

9.2 Data Storage and Handling


All data operations in Synora Health are performed asynchronously using Motor's async MongoDB API. The backend follows consistent patterns for data serialization, ObjectId handling, and pagination.


Key Data Handling Utilities:


serialize_doc(): Converts MongoDB documents to JSON-serializable Python dictionaries by converting ObjectId to string and datetime to ISO format strings. Adds a convenient 'id' field alongside '_id'.


str_to_objectid(): Safely converts string IDs to MongoDB ObjectId with proper error handling for malformed IDs.


paginate_query(): Calculates skip and limit values for MongoDB cursor pagination based on page number and limit parameters.


find_doctor_for_user(): Searches for doctor profile by user_id first, then falls back to email-based lookup for seeded doctors without user_id, auto-linking on first match.


Data Storage Operations:

  • User Registration: OTP stored in otp_records with 10-minute TTL, user created in users collection after OTP verification
  • Appointments: Created with status "pending", updated to "confirmed/cancelled/completed" by doctor or admin
  • Health Records: Stored with patient_id reference, optional file URLs for uploaded documents
  • Chat Messages: Stored in messages collection with room_id for WebSocket room management
  • Payments: Order created in payments collection, verified via HMAC-SHA256 before appointment confirmation
  • AI Analysis Results: Stored within health_records documents as nested objects

Security in Data Handling:

  • All passwords are bcrypt-hashed before storage (never stored in plain text)
  • JWT tokens are stateless — no server-side session storage required
  • OTP codes expire after 10 minutes and are deleted after successful verification
  • Payment verification uses HMAC-SHA256 to prevent payment tampering
  • All database queries use parameterized filters to prevent injection

9.3 Queries and Logic


Authentication Logic:

Registration uses a two-step OTP flow: user data is temporarily stored in otp_records, OTP is sent via Twilio SMS (E.164 format) with email fallback, and the user account is created in users collection only after OTP verification. Login checks credentials, auto-verifies users with correct passwords who were not OTP-verified, and merges doctor profile data into the login response.


Appointment Booking Logic:

Booking involves: (1) fetching doctor availability from doctors collection, (2) checking for conflicting appointments on the selected date+time, (3) creating a Razorpay order via SDK, (4) receiving payment verification from frontend, (5) HMAC-SHA256 signature verification, (6) creating the appointment record on verified payment, (7) sending WebSocket notification to doctor.


Doctor Availability Logic:

Availability is stored as an array of objects: [{day: "Monday", slots: ["09:00", "10:00"]}]. Backend validates that at least one working day with slots exists before saving. Frontend maps available days to enable date picker dates and displays slots as selectable chips.


AI Symptom Analysis Logic:

Symptom data (description, body area, duration, severity, history) is formatted into a structured medical prompt. The prompt is sent to Groq API (Llama 3.1-70B) for analysis. Falls back to Google Gemini if Groq fails, then OpenAI GPT-4 as final fallback. Response is parsed and returned as structured JSON with conditions, urgency, and recommendations.


Chat WebSocket Logic:

FastAPI WebSocket routes manage connection pools per room_id. Messages are persisted to MongoDB immediately and broadcast to all connected clients in the room. A REST endpoint GET /chat/rooms enables notification polling by checking last message timestamps to detect unread messages across all chat rooms.


Admin Stats Logic:

Admin statistics are computed on-demand via async MongoDB count_documents() calls: total_patients (users with role=patient), total_doctors (all doctors), total_appointments, pending_appointments, verified_doctors, unverified_doctors. User and doctor lists support pagination with limit=100 default.

14. LIMITATIONS


14.1 Current Constraints of the System


While Synora Health delivers comprehensive AI-powered healthcare management, the current development version has several limitations that are acknowledged for future improvement:


1. No HTTPS / SSL in Development: The platform runs on HTTP in the development environment. Mobile browsers block geolocation and some camera features on non-HTTPS origins. This limits the Nearby Places and camera-based report scanning features on mobile browsers without HTTPS deployment.


2. AI Responses Not Medically Certified: The AI Health Copilot and Symptom Checker responses are generated by Large Language Models (Groq Llama 3.1, Google Gemini) and are for informational purposes only. They are not a substitute for professional medical diagnosis. A mandatory disclaimer is displayed with all AI outputs.


3. Local-Only Deployment: Currently running on localhost. Cloud deployment (Railway for backend, Vercel for frontend, MongoDB Atlas for database) has not been completed for this version, limiting public internet accessibility.


4. Video Consultation on LAN Only: WebRTC peer-to-peer video calling works reliably on local networks. Internet-based video calls require STUN/TURN server configuration for NAT traversal, which has not been implemented in this version.


5. No Real-Time AI Analysis Cost Monitoring: API call costs to Groq, Gemini, and OpenAI are not tracked or limited per user, which could lead to unexpected costs in a production environment.


6. Limited AI Model Control: The system uses free-tier API access for Groq and Gemini. Rate limits could affect availability during high-usage periods.


7. No Push Notifications: Notification delivery relies on polling (30-second intervals) rather than true push notifications. This means there can be up to a 30-second delay in notification delivery.


8. Payment in Test Mode: Razorpay is configured in test mode during development. Live payment processing requires additional KYC verification and business account setup with Razorpay.
