13.2 Observations

Several important observations were made during testing and execution of the system. Some of these are:

Observation 1: Accurate AI Symptom Analysis

The AI Symptom Checker consistently generated medically relevant results. The Groq Llama 3.1-70B model correctly identified urgency levels (LOW, MEDIUM, HIGH) and recommended appropriate specialist types based on the entered symptoms in all tested cases.

Observation 2: Reliable Payment Verification

The Razorpay HMAC-SHA256 signature verification successfully detected and rejected tampered payment signatures in all security test cases. No fraudulent payment was allowed to confirm an appointment, demonstrating high payment security reliability.

Observation 3: Personalized Doctor Recommendations Improved User Experience

Different patients received different doctor recommendations based on their symptoms and selected specialties. The filtering and matching logic correctly prioritized verified, highly-rated doctors with available slots, improving the booking experience.

Observation 4: Validation Improved System Reliability

The system prevented invalid or unrealistic inputs from reaching the database. Examples:

  • Empty symptom inputs were rejected with proper error messages
  • Invalid appointment slots were blocked from double-booking
  • Weak or incorrect passwords were rejected at login
  • Unauthorized role access was blocked with 403 Forbidden response

Observation 5: Real-Time Features Improved Accessibility

The WebSocket-based real-time chat and WebRTC video consultation successfully connected patients and doctors instantly. Users were able to communicate, share screens, and exchange messages without any significant delay, improving overall healthcare accessibility.

Observation 6: AI Fallback Chain Improved Reliability

When the Groq AI API was temporarily unavailable during testing, the system automatically switched to Google Gemini 1.5 Pro within milliseconds, ensuring continuous AI service availability without any user-visible interruption.

14. LIMITATIONS

14.1 Current Constraints of the System

Even though the "Synora Health AI Platform" effectively provides intelligent healthcare services, AI-based symptom analysis, real-time doctor consultations, and digital health management, there are still some restrictions and limitations with the current version of the system.

1. Dependency on Third-Party AI APIs

The AI symptom checker, prescription assistant, and health record analysis features depend entirely on external AI service providers such as Groq, Google Gemini, and OpenAI.

Limitations:

  • AI features may be unavailable if all three APIs experience simultaneous downtime
  • API rate limits may restrict the number of AI requests per day
  • Future pricing changes by AI providers may affect operational costs

2. AI Diagnosis is Not a Medical Replacement

The AI Symptom Checker provides medically relevant suggestions but is not a certified diagnostic tool.

Limitations:

  • AI outputs are based on training data and may not cover all rare conditions
  • Individual patient history and genetic factors are not fully considered
  • Predicted conditions may differ from an actual doctor's clinical diagnosis
  • Users may over-rely on AI suggestions instead of consulting doctors

3. Local File Storage for Health Records

In the current version, uploaded health record files are stored on the local server filesystem instead of cloud storage.

Limitations:

  • Not scalable for large numbers of users uploading large files
  • Risk of data loss if the server storage fails without backups
  • Geographic distribution and CDN delivery are not supported
  • Enterprise-level deployment requires AWS S3 or Google Cloud Storage

4. Internet Dependency for All Features

All features of Synora Health require an active internet connection to function.

Features depending on internet:

  • AI symptom analysis (Groq / Gemini API calls)
  • Razorpay payment processing
  • Twilio OTP SMS delivery
  • Real-time WebSocket chat and notifications
  • WebRTC video consultation

This limits accessibility for users in areas with poor or no internet connectivity.

5. WebRTC Video Quality on Poor Networks

The video consultation feature uses WebRTC peer-to-peer technology which depends heavily on internet connection quality.

Limitations:

  • Poor network conditions cause video lag and choppy audio
  • WebRTC connections may fail behind strict firewalls or NAT environments
  • TURN server infrastructure is not yet implemented for difficult network scenarios
  • Minimum 5 Mbps internet speed is required for stable video quality

6. No Native Mobile Application

Synora Health is currently a web-based platform and does not have a dedicated native Android or iOS application.

Limitations:

  • Mobile browser experience is less optimized than a native app
  • Push notifications for appointments and medicine reminders are not available
  • Camera and microphone access may be limited in mobile browsers
  • Offline functionality is not supported without a native app

7. Single-Server Deployment Scalability

The current deployment uses a single Uvicorn server instance without load balancing or containerization.

Limitations:

  • Not ideal for very large user bases with high concurrent traffic
  • Limited auto-scaling capability without Docker and Kubernetes setup
  • Enterprise-level deployment requires distributed server architecture
  • No automatic failover or redundancy in case of server failure

8. English Language Support Only

The current version of the platform supports only the English language interface and AI inputs.

Limitations:

  • Patients who prefer regional languages cannot use the platform effectively
  • AI symptom input is limited to English text only
  • This reduces accessibility for rural patients across India
  • Multi-language support requires significant additional development effort
