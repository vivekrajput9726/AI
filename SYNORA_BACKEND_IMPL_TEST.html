CHAPTER 6 — BACKEND DESCRIPTION

6.1 Overview of the Backend Architecture

The backend of Synora Health is built using FastAPI, a modern, high-performance Python web framework based on standard Python type hints. FastAPI was selected for its asynchronous capabilities, automatic OpenAPI documentation generation, built-in data validation using Pydantic, and superior performance compared to traditional frameworks like Django or Flask. The backend follows a layered architecture comprising routes (controllers), services (business logic), models (data schemas), and database interaction layers.

The database layer uses MongoDB, a NoSQL document database, accessed through Motor — an asynchronous Python MongoDB driver that enables non-blocking database operations. This combination of FastAPI and Motor allows the backend to handle thousands of concurrent connections efficiently without blocking threads, which is critical for a healthcare platform with real-time features such as chat, video signalling, and live notifications.

6.4 API Route Modules

6.4.1 Authentication Routes (auth.py)

The authentication module handles all user identity operations. It provides endpoints for user registration with role selection (patient or doctor), OTP verification via Twilio SMS, login with JWT token generation, automatic token refresh, and password reset. The module uses bcrypt with 12 salt rounds for password hashing, ensuring that stored passwords are computationally expensive to crack even if the database is compromised.

from fastapi import APIRouter, Depends, HTTPException
from app.database.connection import get_db
from app.services.auth_service import (
    register_user, login_user, verify_otp, refresh_token)

router = APIRouter()

@router.post("/register")
async def register(data: RegisterSchema, db=Depends(get_db)):
    return await register_user(db, data)

@router.post("/login")
async def login(data: LoginSchema, db=Depends(get_db)):
    return await login_user(db, data.email, data.password)

@router.post("/verify-otp")
async def verify(data: OTPSchema, db=Depends(get_db)):
    return await verify_otp(db, data.phone, data.otp)

@router.post("/refresh")
async def refresh(data: RefreshSchema, db=Depends(get_db)):
    return await refresh_token(db, data.refresh_token)

6.4.2 AI Routes (ai_routes.py)

The AI routes module exposes endpoints for the symptom analysis engine. It receives symptom descriptions in natural language, forwards them to the AI service layer which implements a three-tier fallback chain (Groq → Gemini → GPT-4), and returns structured JSON responses with possible conditions, urgency level classification, and recommended specialist types.

6.4.3 Payment Routes (payments.py)

The payment module integrates with Razorpay's payment gateway. It exposes two primary endpoints — order creation and payment verification. The verification endpoint performs HMAC-SHA256 cryptographic signature validation to ensure payment authenticity before confirming an appointment in the database.

6.4.4 WebSocket Chat (chat.py)

The chat module implements real-time messaging using WebSocket connections. The ConnectionManager class maintains a dictionary of active connections per chat room. When a message is sent by any participant, it is broadcast to all other connections in the same room and persisted to the MongoDB messages collection simultaneously.

class ConnectionManager:
    def __init__(self):
        self.active: dict[str, list[WebSocket]] = {}

    async def connect(self, ws: WebSocket, room_id: str):
        await ws.accept()
        self.active.setdefault(room_id, []).append(ws)

    async def broadcast(self, room_id: str,
                        message: str, sender: WebSocket):
        for connection in self.active.get(room_id, []):
            if connection != sender:
                await connection.send_text(message)

6.5 Database Connection Module

The database connection is managed through Motor's async client. A single global client instance is created at application startup and closed gracefully at shutdown, following FastAPI's lifespan context manager pattern. The get_db() dependency function is injected into all route handlers that require database access.

from motor.motor_asyncio import AsyncIOMotorClient
from app.config.settings import settings

client: AsyncIOMotorClient = None

async def connect_db():
    global client
    client = AsyncIOMotorClient(settings.MONGODB_URL)

async def get_db():
    return client[settings.DATABASE_NAME]

async def disconnect_db():
    client.close()

6.6 Middleware and Security

The backend implements several layers of security and middleware. CORS (Cross-Origin Resource Sharing) middleware is configured to allow requests only from the registered frontend origins. JWT authentication is enforced on all protected routes via a get_current_user dependency that validates the Bearer token in the Authorization header. Role-based access control is implemented through additional dependencies (require_admin, require_doctor) that check the user's role claim within the JWT payload. Rate limiting via SlowAPI prevents brute-force attacks on authentication endpoints.

7.2 Implementation of Key Modules

7.2.1 User Registration and OTP Implementation

The registration process was implemented as a two-step flow. In the first step, the user submits their personal details (name, email, phone, password, role). The backend validates the input, hashes the password using bcrypt, generates a 6-digit OTP, and dispatches it via Twilio SMS to the user's phone number in E.164 format. The OTP record is stored in MongoDB with a 10-minute expiry. In the second step, the user enters the OTP, which is validated against the stored record, and the account is marked as verified.

7.2.2 JWT Authentication Implementation

Authentication was implemented using JSON Web Tokens (JWT). Upon successful login, the backend generates two tokens: an access token (valid for 60 minutes) containing the user's ID, role, and expiry claim, and a refresh token (valid for 7 days). The frontend stores both tokens in Redux Persist (localStorage-backed). The Axios interceptor automatically attaches the access token to every API request and transparently refreshes it when a 401 Unauthorized response is received, providing a seamless user experience without manual re-login.

Figure 7.3: Login Page [ Screenshot: Login page showing Email, Password fields, Quick Demo Login buttons (Patient/Doctor/Admin), and Sign In button ]

7.2.3 AI Symptom Checker Implementation

The AI Symptom Checker was implemented as a three-tier fallback system. The primary engine is Groq's Llama 3.1-70B model, which offers extremely fast inference through Groq's custom LPU hardware. If the Groq API is unavailable or returns an error, the system automatically falls back to Google Gemini 1.5 Pro. If Gemini is also unavailable, a final fallback to OpenAI GPT-4 is triggered. All three models receive the same structured prompt requesting JSON-formatted output with possible conditions, urgency level, recommended actions, and specialist type.

7.2.4 Appointment Booking and Razorpay Payment Implementation

The appointment booking flow was implemented across multiple steps. First, the patient searches for doctors by specialty using the GET /api/doctors endpoint. After selecting a doctor and time slot, the frontend calls POST /api/payments/create-order which creates a Razorpay order via the Razorpay SDK. The Razorpay JavaScript checkout modal opens on the frontend for the patient to complete payment. Upon successful payment, the callback provides a payment ID and signature, which are sent to POST /api/payments/verify for HMAC-SHA256 cryptographic verification before the appointment is saved.

7.2.5 Video Consultation Implementation

Video consultation was implemented using the WebRTC API with a custom signalling mechanism over WebSocket. When both patient and doctor join a video room (identified by the appointment ID), the frontend initiates WebRTC peer connection establishment. ICE candidates and session description protocols (SDP offer/answer) are exchanged via the WebSocket signalling server. Once the peer connection is established, live video and audio streams are displayed in separate video elements on the screen. The room also includes an integrated text chat panel for supplementary communication.

7.2.6 Health Records and AI Analysis Implementation

The health records module was implemented to allow patients to upload medical documents in PDF, JPEG, or PNG format. The file is received by the FastAPI backend using python-multipart for multipart form data handling, saved to the server filesystem using aiofiles for async file I/O, and then passed to the Gemini Vision API for intelligent analysis. Gemini Vision processes the image and returns structured insights including identified medical parameters, abnormal values, and health recommendations.

The admin dashboard was implemented as a single-page application section that fetches real-time statistics from MongoDB at load time. The dashboard displays clickable metric cards (total users, doctors, appointments, revenue) that navigate to corresponding management panels. The doctor verification panel shows pending doctors with their submitted credentials, allowing the admin to verify or reject their registration with a single click. All user management operations (activate, deactivate, delete) are processed through protected admin-only API endpoints.
