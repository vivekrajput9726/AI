18.1 Additional Data, Extended Code, or Supporting Material

This appendix section contains additional technical details, supporting information, extended functionalities, and implementation references related to the "Synora Health AI Platform". It provides supplementary material used during the development, testing, and execution of the system.

Additional Health and Medical Parameters Used

The system uses multiple health and medical parameters for AI analysis and patient management, including:

  • Patient name, age, gender, and blood group
  • Reported symptoms and symptom duration
  • Medical history and existing conditions
  • Allergies and current medications
  • Uploaded health records and lab report values
  • Doctor specialization and consultation fee
  • Appointment date, time slot, and status
  • Payment amount and transaction references

These parameters are used for:

  • AI symptom analysis and urgency classification
  • Doctor recommendation and matching
  • Health record AI analysis and insight generation
  • Medicine reminder scheduling
  • Emergency SOS location-based services
  • Patient health score calculation

Artificial Intelligence and AI Model Support

The project uses multiple AI and Large Language Models (LLMs) with a three-tier fallback chain:

  • Groq Llama 3.1-70B (Primary AI Engine)
  • Google Gemini 1.5 Pro (Secondary / Fallback)
  • OpenAI GPT-4 (Third-level Fallback)
  • Gemini Vision API (Medical image analysis)

The AI modules analyze patient health data and generate:

  • Possible medical conditions from symptoms
  • Urgency level (LOW / MEDIUM / HIGH)
  • Recommended specialist doctor type
  • Suggested actions and follow-up steps
  • AI-assisted prescription drug suggestions for doctors
  • Smart health record analysis and insights

The recommendation engine also performs:

  • Doctor matching based on patient symptoms
  • Appointment slot availability checking
  • Payment HMAC-SHA256 signature verification
  • Health score trend analysis

Additional Security Features

The project includes several security mechanisms to protect user data and system integrity:

  • Secure JWT-based authentication with role-based access control
  • bcrypt password hashing with 12 salt rounds
  • Twilio SMS OTP verification for account registration
  • HMAC-SHA256 cryptographic signature verification for payments
  • CORS (Cross-Origin Resource Sharing) protection
  • API rate limiting to prevent brute-force attacks
  • Input validation using Pydantic schema models
  • Admin-only access control for sensitive operations
  • Refresh token rotation and invalidation on logout
  • MongoDB query injection prevention through Motor driver

These features improve system reliability, prevent unauthorized access, and ensure user data security throughout the platform.

Database Support Information

MongoDB (NoSQL) database is used for storing all platform data. The following collections are maintained:

  • User records (patients, doctors, admins)
  • Doctor profiles and availability schedules

  • Health records and AI analysis results
  • Real-time chat messages and room data
  • OTP records with expiry timestamps
  • Medicine reminder configurations
  • Emergency SOS logs
  • Notification records
  • Health goals and symptom diary entries

Additional Testing Support

The system was tested using multiple scenarios to ensure robustness and reliability:

  • Login with correct and incorrect credentials
  • Invalid OTP and expired OTP submissions
  • Booking an already-taken appointment slot
  • Tampered Razorpay payment signature
  • AI API failure with fallback chain activation
  • Unauthorized access to admin-only endpoints
  • Patient using doctor-only routes (role mismatch)
  • Uploading invalid file types for health records
  • WebSocket disconnection and reconnection
  • Mobile browser access via LAN IP address
  • Token expiry and automatic refresh flow
  • High concurrent API requests (load simulation)

The system successfully handled all these test cases using proper validation, error handling mechanisms, and fallback strategies.
